Book Club

A shared reading room for small groups: one current book with a meeting date and discussion notes, a queue of suggestions, and an archive of everything the club has finished.

The problem

Small book clubs run on scattered group chats: "what are we reading again?", "when do we meet?", and half-remembered suggestions that get lost between meetings. Thoughts people wanted to bring to the discussion evaporate before the gathering. General-purpose tools (Goodreads groups, chat threads) are either too heavy or too lossy for a five-person club.

The solution

Book Club is the third Rooms template. The club is a room: members join with the invite code, everyone sees the same current book, meeting date, queue, and notes — synced local-first (Yjs CRDT on-device, end-to-end encrypted relay), so the club's reading history and discussion notes live on members' own phones, not a service.

Key flows

  • Create & join — pick Book Club, name the club, add members (minimum 2). Share the room code; each device picks "who's on this device" via the profile picker.
  • Current book & progress (Reading tab) — exactly one book can be reading at a time. It carries an editable meeting date for the next discussion. Members add text notes attributed to them (newest first) to bring to the meetup. There is no page-count or "% read" tracking in v1 — progress is the book's status plus the meeting date.
  • Queue (Queue tab) — anyone suggests upcoming titles (title + author, attributed via addedById). Queued books are ordered first-suggested-first. Anyone can tap "Start reading this" to promote a queued book to current — there is no voting in v1; promotion is first-come.
  • Archive (Archive tab) — finished books, newest finish first. Books arrive here two ways: explicitly via Mark done, or automatically — starting a new book auto-archives whatever was being read.
The archive flow in one line startReading(bookId) walks all books in one transaction: any other book with status reading is flipped to done (stamped finishedAt, meeting date cleared), then the target becomes reading. This is what guarantees the single-current-book invariant even when two members promote different books concurrently — the CRDT merges and the later write wins, with the loser ending up archived rather than duplicated.

Data model (Yjs keys)

Public branch only — no admin-only club data.

KeyHolds
template.bookclub.club{ name, createdAt }
template.bookclub.membersMap of { id, name, color, joinedAt }
template.bookclub.booksMap of { id, title, author, status, meetingDate?, addedById, addedAt, finishedAt? } — status is reading | queued | done
template.bookclub.notesMap of { id, bookId, authorId, body, createdAt }

Notable rules & edge cases

  • The "one current book" rule is enforced by the write path (startReading), not a schema constraint; getCurrentBook() just returns the first reading book.
  • Marking a book done or replacing it clears its meetingDate — meeting dates only belong to the active read.
  • Notes are keyed by bookId and survive archiving in the data, but the UI only shows notes on the current book — archived titles don't display their old notes in v1.
  • Books added with a status can start life directly as reading (used at setup) instead of going through the queue.
  • Intentional v1 limits: no queue voting, members added at setup only, no page progress tracking, no editing/deleting notes.

  • apps/rooms/web/src/templates/bookclub/BookClubApp.tsx — shell: loading → setup → profile → main.
