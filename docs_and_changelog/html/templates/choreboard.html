ChoreBoard

A family chores-and-allowance room: parents publish a priced chore catalog, kids mark chores done, and approvals roll up into a running balance that gets settled on payday.

The problem

Turning kids' chores into a fair, motivating allowance is messy. There's rarely a shared, agreed list of which chores exist or what each is worth, kids can't watch their money grow, parents lose track of what was actually done during the week, and payday becomes guesswork ("how much do I owe you?"). Paper charts get erased, spreadsheets are joyless, and existing allowance apps are cloud SaaS that hold your family's data with subscriptions and ads.

The solution

ChoreBoard is the first Rooms template: a local-first, phone-friendly chore + allowance tracker where the family owns its data. Everything lives in Yjs CRDT documents on the family's own devices, synced end-to-end encrypted through the thin relay — no server ever sees chores, balances, or kids' names.

A note on history A standalone predecessor of this app exists at choreboard/ in the repo root (its own web app and relay). It was folded into the Rooms platform; this page documents the current template at apps/rooms/web/src/templates/choreboard/, which is the live version.

Roles: parent vs kid

Every member has a role of parent or kid. The split is enforced with cryptography, not just UI — there are three secrets, each unlocking a different power:

SecretWho has itWhat it unlocks
Family codeEvery devicePublic channel: members, published catalog, completions, kid proposals
Parent secretParents onlyAdmin channel: edit chores, payments, settings, permission policy
Member device linkOne kid profile per deviceHMAC-signs that kid's completions so a parent can verify them

The room has two encrypted relay channels. Kids holding only the family code never receive the admin channel's ciphertext, so they mathematically cannot edit chore prices, approve completions, or run payday — even with a modified client. All secrets are 16 random bytes (~128 bits), not guessable word phrases.

Permissions (what kids see)

On top of the crypto split, parents configure per-kid visibility permissions (Settings → Kid permissions): see chore rewards, see own/pending/week balance, see sibling balances (off by default), mark done, suggest chores, and see activity history. Defaults plus per-kid overrides are authored on the admin doc and published to the public doc so kid devices can apply them. Honest limit: these are UI policy on data already in the public sync doc — a modified client could ignore them. Member device-link signatures are the stronger accountability tool.

Key flows

  • Setup — a parent creates the room, names the family, sets currency (default USD) and payday weekday, adds members, and gets a seeded chore catalog to edit.
  • Kid daily use (KidView) — pick your profile, see the published catalog with difficulty and cadence pills, tap ✓ to mark done. A "balance hero" shows money so far, pending approval, and this-week earnings (each gated by permissions). Kids can also suggest new chores.
  • Approvals — chores with requiresApproval create pending completions; ones without are auto-approved (approvedBy: "auto"). Parents approve or reject from the Approvals tab, which also surfaces kid chore proposals — approving a proposal promotes it into the catalog (with approval required and the default 1/day limit).
  • Penalties — parents can add a negative completion (e.g. "left bike outside, −$1"); it nets against the balance automatically.
  • Payday — the parent dashboard shows what's owed per kid (sum of approved completions). "Mark paid" snapshots them into a Payment record on the admin doc and flips those completions to paid.
Parent dashboard layout (current, includes uncommitted changes) ParentView.tsx currently uses the new shared BottomNav with five tabs: Home (total owed + per-kid cards), Approvals (with a pending + proposals badge), Chores (catalog CRUD via ChoreForm), Payday, and Settings (members, permissions, invites/QR, relay, reset history). This BottomNav switch is an uncommitted work-in-progress change.

Chore frequency limits

Kids can only mark a chore done a limited number of times per period, which stops repeat-tapping from stacking rewards. Parents set "Times per day/week/month/ever" (or no limit) on each chore; new chores default to once per day. Rules that matter:

  • Enforced in store logic (ChoreStore.markDone checks canMarkChoreDone), not just by disabling the button.
  • Pending completions count toward the limit; rejected ones free the slot; paid ones still count within the current period.
  • Legacy chores without a frequencyLimit fall back to the old recurrence field (daily → 1/day, weekly → 1/week, one-off → once ever, anytime → unlimited).
  • Kid UI disables the ✓ and shows status like "Done for today" or "2/3 this week".

Payments & ledger

The ledger is the completions map: every entry has a signed amount snapshot (positive reward, negative penalty) captured at completion time, so later price edits never rewrite history. Derived numbers are pure reductions over it: balance = sum of approved amounts, pending = sum of pending, this week = approved + paid within the Monday–Sunday range. markPaid creates a Payment with the period (earliest approved date → today), total, and the exact completion ids it covers — an audit trail on the parent-only admin doc. Parents can also "Reset history" (wipes completions, proposals, payments; keeps chores, members, settings).

Data model (Yjs keys)

Everything lives under the template.choreboard branch, split across the two docs:

DocKeyHolds
Publicfamilyname, currency, payday weekday, createdAt
membersMap of { id, name, role, color, pin?, createdAt }
catalogPublished copy of active chores (read-only for kids)
completionsMap of { choreId?, label, memberId, date, kind, status, amount, sig?, approvedBy? }
proposalsKid chore suggestions awaiting a parent
permDefaults / permOverridesPublished kid-permission policy
AdminchoresSource-of-truth catalog incl. archived; edits republished to public catalog
paymentsPayday records { memberId, periodStart, periodEnd, total, completionIds[], paidDate }
family, permDefaults, permOverridesAdmin-authored mirrors, published to public on change

The publish step (publishCatalog / publishPermissions) copies admin truth onto the public doc whenever a parent edits — that's how kids see only active chores and current policy without ever holding the admin key.

Notable rules & edge cases

  • One price per chore, the same for every kid (locked v1 decision).
  • Unsigned completions are allowed (e.g. a shared tablet without a device link); parents see a verification warning instead of a hard block. Penalties and auto-approved completions skip signature checks.
  • Chores are archived, not deleted, so historical completions keep their labels (the label is also snapshotted on each completion).
  • Week math is Monday–Sunday regardless of the configured payday weekday; payday weekday is informational for the dashboard.
  • Kid PINs are an optional profile-picker convenience on shared devices, not a security boundary.
