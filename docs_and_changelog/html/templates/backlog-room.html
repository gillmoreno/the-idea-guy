Backlog room

A community idea pool for what to build next — friends propose and vote inside a room, and the owner decides what ships.

The problem

The Idea Guy ships small apps for friend groups, and the obvious question is "what should we build next?" Collecting that input normally means a feedback board product or a database with accounts — exactly the kind of central infrastructure the rest of the product avoids.

The solution: a room, not a database

The Backlog is itself a Rooms template, built on the same stack as ChoreBoard, Trip Split, and Book Club:

  • A Yjs CRDT document on each participant's device.
  • End-to-end encrypted sync through the dumb relay — no central server ever reads the ideas.
  • An invite code gives "public-ish" participation without Postgres or accounts.
Why this matters The product story is local-first, and the Backlog proves it applies even to community input. Anyone with the room code can join, propose, and vote — and the only authority the owner keeps is the status of each idea.

Key flows

  1. Create — pick the Backlog template, name the idea pool, set up voters. Three starter ideas are seeded automatically (see below).
  2. Share — friends join with the room code and pick their voter profile (no accounts).
  3. Propose — anyone adds an idea: title, emoji, short pitch. New ideas start as proposed.
  4. Vote — one vote per person per idea, toggled on/off with a single button. The list re-sorts live by vote count, then recency, and the top idea earns a #1 pill.
  5. Owner status control — only the room owner sees a status dropdown per idea and moves it through proposed → building → shipped → parked. Everyone else sees the status as a read-only pill on the idea card.

Seeded ideas (v1)

On first setup, initBoard pre-loads three extension-template proposals (attributed to the organizer) so the room is never empty:

EmojiIdeaPitch
🏠💡Roommate LedgerOngoing household bills and shared costs — Trip Split for daily life, not just vacations.
🍿📺Watch ClubBook Club for TV — current series, episode notes, queue the next binge.
🥗🍷PotluckWho brings what, dietary tags, and headcount for shared meals.

Users can add more proposals anytime; seeds are ordinary ideas with no special treatment.

Data model (template.backlog.*)

KeyShapeHolds
template.backlog.boardMap{ name, createdAt } — a set name marks the room as initialized.
template.backlog.membersMap<id, Member>{ id, name, color, joinedAt } — the voters.
template.backlog.ideasMap<id, BacklogIdea>{ title, description, emoji, status, proposedById, proposedAt }.
template.backlog.votesMap<ideaId, Map<memberId, votedAt>>One nested Y.Map per idea; a member's presence in it is their vote.

Votes deliberately live in their own nested map rather than as a counter on the idea: presence/absence per member merges cleanly under CRDT semantics (two devices toggling concurrently can't double-count), and "one vote per person" falls out of the map key.

Notable rules and edge cases

  • Votes are a toggle — toggleVote deletes your entry if present, otherwise sets it with a timestamp. The vote count is just the nested map's size.
  • Sorting is computed, not stored — listIdeas sorts by vote count descending, then proposedAt descending. Nothing persists ranking, so it is always consistent with the votes.
  • Status is owner-only by UI, not cryptography — the dropdown renders only when isOwner is true. Suitable for a friendly idea pool; it is a convention, not an enforcement boundary.
  • Setup gating — until the owner initializes the board, joiners see a "connecting" screen; afterwards, anyone without a member profile goes through the profile picker.
  • Invite slots become voters — reserved invite slots are pre-created as members so invited friends land directly on their profile. Invites are optional (minContacts: 0); sharing the raw room code works the same.
  • Single Yjs instance rule — the store imports Y from @the-idea-guy/room-kit (never yjs directly); a duplicate Yjs copy broke instanceof Y.Map checks (like the vote-bucket check) and corrupted CRDT state. Rooms created before that fix may need recreating.

Future direction

  • Link shipped ideas to a real template in the registry.
  • Optional read-only "showcase" room code on the marketing site — a public roadmap with no backend.
  • Import/export of the seed list.
