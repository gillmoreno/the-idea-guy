Inkanto

A story-writing coach — not a story writer — built for an 11-year-old novelist: an Italian-first (it/es/en) mobile PWA where the AI asks one good question and offers three options, but she writes every word herself.

What it is

Inkanto ("il tuo laboratorio di storie") is a personal writing lab built for Gil's daughter. She keeps her stories in one place — an idea, an outline of beats (the scaletta), chapters, and a story bible (the mondo: characters, places, objects) — and a Claude-powered coach sits alongside, available in six distinct coaching modes. The core product promise, enforced in the prompts themselves, is that the coach never writes her story. It asks questions, offers numbered options to pick from or remix, celebrates the specific things that make her ideas hers, and keeps everything age-appropriate.

Status: live at inkanto.the-idea-guy.com, family-only by design. There is no signup at all — the register endpoint and the old FAMILY_CODE gate were removed. Accounts come from the INKANTO_USERS env var (user:pass,user:pass): at startup the Go API creates any listed user that doesn't exist yet and never touches existing ones. Production is one Docker container (nginx + Go API + AI sidecar under supervisord) on host :3400, reached through the existing choreboard-relay Cloudflare tunnel (proxied CNAME inkanto → tunnel) — Cloudflare terminates TLS, so the PWA gets its secure origin with no host-level proxy. just dev inside inkanto/ still runs web :3400 / Go API :4800 / sidecar :4700; deploy and aftercare details live in inkanto/DEPLOY.md.

Why it was built this way

Unlike Rooms and Second Brain (local-first Yjs apps), Inkanto follows the classic Roger monolith pattern: a Go chi API with SQLite, a React/Vite SPA served by nginx, all run by supervisord in a single container. A kid's writing app wants one durable source of truth a parent can back up (one SQLite file in ./data/), simple JWT accounts, and zero sync complexity — local-first CRDTs would be over-engineering here.

The interesting choice is how the AI is wired. Instead of an Anthropic API key with pay-per-token billing, Inkanto adds a small stateless Node sidecar that uses the Claude Agent SDK authenticated with a Claude subscription OAuth token (CLAUDE_CODE_OAUTH_TOKEN, minted once via claude setup-token). Coach calls draw from the subscription's monthly Agent SDK credit — a family app rides the existing plan with no separate API bill. The sidecar runs the SDK with allowedTools: [] and maxTurns: 1: pure conversation, no file or shell access, one reply per call.

  • Go owns all data. Stories, beats, chapters, entities, and coach history live in SQLite behind the Go API. The sidecar holds no state at all.
  • The sidecar exists only because the Agent SDK is a Node library. Go composes the request; Node just turns it into a streamed Claude reply.
  • One origin for the browser. The frontend only ever talks to the Go API; even the skill catalog is proxied through Go from the sidecar.
  • glibc base image on purpose. The final container is node:20-slim rather than alpine because the Agent SDK ships a native claude binary that isn't guaranteed to run on musl.

The skills system

Coaching behavior is not hardcoded — it lives in inkanto/skills/*.md, Claude-Code-style markdown files with YAML frontmatter (name, emoji, localized titles, description). The sidecar loads them at startup and builds each system prompt as base guardrails + selected skill + locale note. Editing a coaching mode means editing a markdown file and restarting the sidecar.

_base.md is the always-on guardrail layer, prepended to every session and hidden from the skill picker: reply in her language (Italian by default), never write her story (at most one short marked example sentence), prefer "one question plus 3 numbered options", keep replies short, stay age-appropriate for an 11-year-old, never ask for or discuss personal real-life information, and shrink the task when she's stuck.

What users see

  • Login only — no signup. The login page is a redesigned animated "enchanted book cover" (floating sparks, a quill, a shake on a wrong password). Accounts are provisioned by a parent via INKANTO_USERS; sessions are plain JWT bearer tokens.
  • Stories with structure. Each story has an idea, a scaletta of beats, chapters with statuses, and a mondo of typed entities (characters / places / objects with summaries and notes).
  • A coach drawer per story. She picks a skill, and each skill keeps its own conversation history (the last 50 turns are shown). Replies stream token-by-token over SSE, so the coach feels alive on a phone.
  • Context-aware coaching. Asking the coach from inside a chapter or an entity sends that chapter's text (last ~6,000 characters) or that entity's notes along as the focus.

Data flow of a coach call

One request travels browser → Go → sidecar → Claude and streams back the whole way:

  1. The frontend POSTs /api/stories/{id}/coach with a skill, message, and optional chapter/entity focus.
  2. Go verifies ownership, then flattens everything the coach should know into one text block: title, idea, scaletta, mondo, chapter list, plus the focused chapter or entity.
  3. Go attaches the last 12 turns of that skill's history and the writer's locale, and POSTs to the sidecar's /coach.
  4. The sidecar folds context + history + message into a single-turn Agent SDK query() with the composed system prompt, and streams text chunks back as SSE.
  5. Go relays the SSE lines to the browser unchanged while accumulating the full reply, then persists both the user turn and the assistant turn. The user turn is only saved after the sidecar accepts the request, so a dead sidecar never pollutes the history.

Tradeoffs

  • Two runtimes instead of one. The Node sidecar adds a second process to deploy and supervise, but it's the price of subscription auth via the Agent SDK — and supervisord already runs nginx + Go anyway.
  • Manual token rotation. The OAuth token lasts about a year and does not auto-rotate; when it expires someone re-runs claude setup-token and updates .env.
  • Credit, not quota. The Agent SDK draws from the subscription's monthly Agent SDK credit (separate from interactive Claude Code limits since June 2026). If it runs out, the coach returns an error — but the writing app keeps working fully; the AI is additive, never load-bearing.
  • Guardrails are prompts, not code. The "never writes her story" rule lives in _base.md rather than an output filter. That keeps it editable and inspectable by a parent, at the cost of being model-behavior rather than a hard technical guarantee.
  • Accounts need a restart. With no signup endpoint, adding a user means editing INKANTO_USERS in .env and restarting the container. For a family-sized userbase that's a feature: no registration surface exposed on a public URL, and the user list is one greppable line. (bootstrapUsers enforces ≥4-char passwords, lowercases usernames, and defaults display name to the capitalized username with locale it.)
