Sync & local-first

Room data lives on users' devices as encrypted Yjs CRDT documents; a dumb relay only ferries opaque blobs between peers, so the app works offline, syncs conflict-free, and nobody — including the operator — can read anyone's data.

The core principle

Serving the app is not the same as storing the data. Serving static HTML/JS from a CDN is nearly free and scales to thousands of users. Storing user data is the expensive, liability-heavy part — so Rooms simply doesn't centralize it. The app is distributed centrally and cheaply; the data stays owned by the people who created it. That's the local-first philosophy, and it doubles as the business model: free hosted tier costs near nothing to run, self-hosting the relay is possible (one small Go binary / Docker image), and a paid managed-sync tier is the only place infra money changes hands.

On the device: Yjs + IndexedDB

Each room channel (public, and admin for owners/admins) is one Yjs CRDT document. CRDTs merge concurrent edits from multiple devices without conflicts — two kids checking off chores offline at the same time both win. The document is persisted locally with y-indexeddb, so the room opens instantly from disk before any network is involved, and keeps working with no connection at all.

The room kit wraps all of this in LocalFirstDoc: it derives the encryption key and relay room id from the invite code, loads the local IndexedDB copy, connects to the relay, and merges whatever arrives. Local edits are encrypted and pushed as they happen; the relay connection can drop and reconnect freely because IndexedDB is always the source of truth on-device.

Offline behavior: everything works. Reads come from IndexedDB; writes are applied to the local doc immediately and queued by Yjs semantics — the next time any relay connection succeeds, the merged state flows both ways. There is no "offline mode" to enter; offline is the default assumption.

End-to-end encryption: the relay is blind

Every Yjs update is encrypted on the client before it leaves the device, with an AES key derived from the room's invite code (plus an optional Argon2id-strengthened passphrase for extra-sensitive rooms). The relay room id is itself a SHA-256 hash, so the relay never even sees the invite code. What the relay stores and broadcasts is pure ciphertext — it has no accounts, no auth, no idea which template a room runs, and no ability to read anything. Access control is the key: if you have the invite code, you can decrypt; if you don't, the blobs are noise. Public and admin channels use different key material, which is how member/admin separation is enforced cryptographically rather than by server logic.

The relay protocol, briefly

The relay (relay/main.go) is a Go WebSocket server with one real endpoint: /sync?room={hash}. Per room it keeps a replay log of encrypted frames in memory (optionally on disk) and broadcasts new frames to connected peers. The wire protocol is a single unencrypted prefix byte on each frame:

FrameDirectionMeaning
0x00 updateclient → relayIncremental encrypted edit — append to the room log and broadcast
0x01 checkpointclient → relayFull merged encrypted state — replace the room log with this one frame
0xFE sync endrelay → clientBacklog replay finished; client may now checkpoint

The relay does zero parsing of ciphertext — a one-byte tag check, a 32 MiB size ceiling, append/broadcast, done. That minimalism is a performance principle as much as a privacy one.

Log compaction

An append-only log would grow forever, and every reconnect used to push a full snapshot. The fix is client-driven checkpoint compaction: after a client replays the backlog and merges it with its local IndexedDB state, it sends one checkpoint frame containing the whole merged document, and the relay collapses the room's log to that single entry. Incremental updates append again until the next checkpoint. Tradeoffs accepted by design: compaction only happens when a device connects (no background GC on the server), and the relay retains no edit history beyond the latest checkpoint — which is a privacy feature. Clients also have a local compactStorage path to prune CRDT history after large payloads like inline images.

Background sync & "Updated" badges — no push

Rooms deliberately has no push notifications and no lock-screen alerts. Instead, when the home screen is open and the app is foregrounded, the shell quietly pulls each vault room: pullRoomPublicChannel opens a short-lived WebSocket, replays the relay backlog into local IndexedDB, sends a checkpoint (helping compaction for free), and disconnects. Up to three rooms sync in parallel; triggers are home mount and the app returning to foreground.

Each room's state is fingerprinted with a SHA-256 hash of its Yjs state vector. If the hash differs from the baseline saved when the user last opened that room, the room card shows an Updated badge. Opening the room clears the badge and refreshes the baseline. Only the public channel is checked in v1 — admin-only edits don't badge. The whole feature is a user toggle (Profile → Check rooms on home, default on).

Why this design: push notifications require a server that knows who you are and what changed — exactly the infrastructure (and liability) this architecture avoids. "Awareness while you're already in the app" gets most of the value at zero infra cost and zero privacy cost. Accepted limits: no semantic summaries ("2 chores done"), no sync while the app is closed, staleness up to one foreground return.

The Yjs single-instance rule

One sharp edge worth knowing: app code and template stores must import Y from @the-idea-guy/room-kit — never from yjs directly. Webpack is aliased so exactly one copy of Yjs is bundled for the app and the room kit. When two copies snuck in, instanceof Y.Map checks failed across bundle boundaries and actually corrupted CRDT state (upstream issue yjs#438); some rooms created before the fix had to be recreated. The kit also exports a duck-typed isYMap helper that stays safe even across duplicate bundles.
