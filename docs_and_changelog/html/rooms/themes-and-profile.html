Themes, profile & settings

How Rooms gets a personal look — four CSS-only themes, a persona accent that recolors the whole app, and a small set of shared settings blocks every template reuses.

The /profile page

Profile (/profile) is the one place for everything about you on this device, separate from any individual room. It holds:

  • Persona — your name and avatar (the avatar brick: an emoji, default 😊, or a square-cropped photo rendered as a circle). The avatar shows in every room header via TopbarPersona, which also links back home.
  • Accent color — a solid color or gradient picked with AccentColorField (curated swatch grid plus native color picker; 16 one-tap gradient presets).
  • Look & feel — the theme picker (ThemeSwitcher), which lives exclusively here since 2026-06-07.
  • App lock — an optional device PIN (see settings below).

The reasoning: room templates should stay focused on the room's job, while identity and appearance live in one predictable place that applies app-wide on the device.

Four CSS-only themes

Themes deliberately use no images — they're pure CSS variables and selectors keyed off data-theme on <html>, so switching is instant and snappy on mobile.

ThemeVibeBottom-nav icon treatment
ClassicClean, minimal baselineFull-color emoji icons
PaperWarm stone neutrals, flat cardsSoftened inactive icons
SignalDot grid, cyan accent, sharp panelsGrayscale inactive, full color on active
GlowDark glass, gradient buttonsFloating glass bar with accent glow on the active tab

The choice is stored in localStorage under rooms.theme.v1 — per device, no server, no sync. loadTheme() falls back to Classic for unknown values and quietly migrates retired theme ids (the old "aurora" theme maps to Paper), so an old stored value never breaks the app. saveTheme() writes both localStorage and the data-theme attribute, so the change applies immediately to every screen, including the bottom navigation's icon style.

Persona accent → --primary

Your accent color isn't decoration on the profile page — it becomes the app's primary color. applyPersonaAccent() sets inline CSS variables on <html> that override whatever the theme defines:

  • --primary — the accent itself (for gradients, the "from" color), driving buttons, links, and highlights.
  • --primary-dark and --primary-soft — derived automatically with CSS color-mix() (78% toward black; 14% mixed into the surface color), so one picked color yields a coherent mini-palette with no manual tuning.
  • --primary-gradient — set only when the accent is a gradient, removed otherwise.

This is why theme and accent compose cleanly: themes define the canvas, the persona accent recolors the interactive layer on top of any of the four.

Template icons

Template emoji strings can contain two glyphs (Book Club is 📚☕). Instead of rendering them as awkward stacked text, TemplateIcon draws an app-style tile: the primary glyph centered on a squircle tinted with --template-accent, and the secondary glyph in a small corner badge. Single-emoji templates get just the tile. splitEmojiGlyphs() parses the graphemes, so the registry's plain emoji field is unchanged. The tile appears on the Create picker, the home room list, loading screens, room invites, and the declarative room top bar; the older single-emoji "orb" survives only on record rows (ideas, prizes).

Emoji picker

One shared picker (EmojiPicker trigger + lazy-loaded EmojiPickerPanel) wraps emoji-picker-react v4: the full Unicode set with native system glyphs, search, category tabs, recents, and skin tones. It theme-follows Rooms — Classic and Paper get the light picker, Glow and Signal the dark one. Consumers today are declarative schema emoji fields and Backlog's "Propose an idea"; anything else that needs an emoji should use the same component rather than a text input.

Settings architecture

Templates don't write their own settings UIs; they compose shared shell blocks, so behavior and copy stay consistent across rooms:

BlockWhat it does
RelaySettingsCustom sync relay URL + reconnect. Hidden by default; the operator enables it at build time with NEXT_PUBLIC_RELAY_SETTINGS_ENABLED (the static export bakes flags in). Aimed at self-hosters and power users.
RoomInviteSettingsInvite mutual contacts into a room. Renders only for users with admin access (admin secret unlocked). Each template passes onReserveMembers so reserved member slots land in its own CRDT before inbox invites go out.
RoomCodeShareRoom code plus join QR.
AppLockSettingsOptional device PIN. Room codes and passphrases sit in localStorage, readable to anyone who opens the browser — the PIN encrypts the entire device vault, with the key derived via Argon2id. No server, no recovery: forget the PIN and you rejoin rooms from invite codes.
TopbarPersona / AppTabBarStandard header with persona avatar, and pill tabs under the topbar (Trip Split, Book Club).

In progress: bottom navigation

A new shared BottomNav component (uncommitted as of this writing, alongside related edits to globals.css, the theme definitions, and ChoreBoard's parent view) adds a fixed bottom tab bar for templates with 3–5 primary sections — ChoreBoard's parent view is the first adopter. Each BottomNavItem takes an emoji icon, label, active state, and an optional numeric badge (e.g. pending approvals). Its icon treatment is theme-driven, which is why the theme taglines now mention nav style: each of the four themes styles inactive/active tabs differently, with tap and active micro-animations defined under .bottom-nav in globals.css. Expect details to shift until this lands.

Tradeoffs

  • Per-device, not synced — theme and accent live in localStorage, so your phone and laptop can look different. Simple and private, at the cost of "set it twice".
  • CSS-only themes — no imagery keeps switching instant and the bundle small, but limits how far apart the themes can feel; the bottom-nav icon treatments are how each theme earns a distinct personality cheaply.
  • App lock has no recovery — consistent with the no-server philosophy: convenience traded away so a forgotten PIN can't be a support backdoor.
