Declarative schema engine

A versioned "Lego bricks" JSON format that lets users (often with an AI chatbot's help) describe a new room type, which a shared engine then renders — no new code, no new deploy.

Why it exists

Every built-in room (ChoreBoard, Trip Split, Book Club, Fit Crew) is hand-written React: a store, views, setup flow, settings. That is the right call for rich apps, but it makes "I want a room for X" a multi-day engineering task. The declarative schema engine flips that: a room type becomes a small JSON document — name, emoji, accent color, collections of records with typed fields, and optional features like votes — and one shared runtime (the SchemaEngine) renders any such document. New templates go from days of coding to minutes of describing.

The pitch in one line Users paste a system prompt into ChatGPT / Claude / Grok, describe the room they want, paste the JSON back into Rooms, hit Validate, and have a working synced room.

Builtin vs declarative rooms

BuiltinDeclarative
Who builds itGil (compiled React per template)User + AI (JSON schema)
RuntimeLazy-loaded app per templateOne shared SchemaEngine
Where the definition livesIn the app bundleInside the encrypted room document (meta.schema)
DistributionCreate → share inviteSame invite flow; schema travels with the synced doc

The relay is unchanged either way — it remains a dumb encrypted-blob pipe and never sees schemas or data. TemplateApp.tsx routes a room to the right runtime based on meta.templateKind ("builtin" or "declarative"). Joiners briefly use a _pending template id until the synced doc reveals the real type.

The schema spec (v1)

A schema is a RoomSchema: id, name, description, emoji, optional accent color, a list of collections (each with typed fields), and optional features. Fields and features are the "bricks".

v1 field bricks (record fields)
TypeStored asUI
textstringSingle-line input; titles on cards
textareastringMulti-line notes / pitch under the title
tagsstring arrayComma-separated input rendered as pills
emojistringShared EmojiPicker (search, categories, recents); shown as a small icon tile on the record
imagestring (JSON)Shared ImageField — upload (auto-crop WebP ≤ 300 KB inline) or public URL
v1 feature bricks (behavior on a collection)
TypeWhat users see
votesPer-member upvote button on each record; the list re-sorts by count
statusAn enum dropdown on records; setBy decides whether only the owner or any member can change it

Related bricks live outside record schemas but follow the same "one brick = one reusable component" philosophy: the accent brick (solid hex or one-tap gradient presets, picked with AccentColorField) and the avatar brick (emoji or square-cropped ≤ 80 KB WebP photo, picked with AvatarField) are used in persona onboarding and the profile page. Candidates earmarked for v2 — number, date, select, money, member fields and assign, dueDate, tally, reactions features — are deliberately not valid in v1 AI output, so the prompt can't generate bricks the engine doesn't render yet.

Versioning & migrations

Schemas live inside long-lived encrypted room documents on people's devices, so the format must evolve without ever breaking an existing room. Two version numbers handle the two directions of change:

  • schemaVersion — the shape of the JSON document. On load, migrate.ts upgrades older schemas in place with additive-only changes (e.g. filling in defaults for fields that didn't exist yet).
  • engineVersion — the minimum app build that understands all the features a schema uses. If a room needs a newer engine than the one running, the user sees a friendly "Update required" screen instead of a broken room.
  • Unknown brick types are skipped, never executed. An older engine that meets a schema with a field or feature type it doesn't know simply ignores it (unknown fields render as read-only text), so new bricks can ship without invalidating old apps.
  • extensions — a reserved object on the schema for future per-room metadata without a version bump.

One accepted tradeoff: editing an existing room's schema after creation is not in v1 — it was judged too complex. Rooms keep working as-is while the engine gains bricks around them.

How the SchemaEngine renders a room

A declarative room stores its full schema at meta.schema in the CRDT, and its live data under template.declarative.* (board name, members, data.{collectionId}.{recordId} records, and per-member vote maps). When the room loads, DeclarativeApp.tsx walks a guard chain:

  • Show loading screens (tinted with the schema's accent) until the local doc is ready.
  • Parse and migrate the schema; show "Update required" if engineVersion is too new.
  • If the room data isn't initialized yet: the owner gets the setup flow, joiners get a "connecting to the organizer" screen.
  • If the device has no member identity yet, show the profile picker.
  • Finally render CollectionView for the schema's collections — RecordCard per record, FieldInput per field type in the add form, vote buttons and status dropdowns where features declare them — under the standard shell (persona top bar, sync badge, invite settings, room code share).

The room's accent becomes a --template-accent CSS variable, so every declarative room gets its own tint with zero custom CSS.

Creating a declarative room

  • Official catalog — curated schemas in public/catalog/v1.json (a marketplace seeded from the Backlog room's best ideas).
  • Paste JSON — Create → Paste JSON → Copy system prompt → describe the room to an AI → paste the JSON back → Validate → Create. validate.ts is the gate: bad JSON never reaches the CRDT; valid schemas are stashed in sessionStorage until the owner's setup writes them into the room doc.
  • One-tap examples — Watch Club and Weekend Plans buttons on the Create screen, for trying it without an AI.

The system prompt itself ships in the app (schema/prompt.ts is the source of truth, mirrored to public/prompts/room-schema-v1.txt) so it can constrain AI output to exactly the v1 brick set.

The QA harness

Because one engine renders arbitrary user schemas, a CSS regression in RecordCard can break every declarative room at once. The defense is an automated UI gate, run with npm run qa:schema-ui (or make schema-qa from apps/rooms, with the dev server running):

  • No vision models — the v1 philosophy is deterministic browser geometry. A Playwright script loads /schema/preview, which renders fixed fixtures (a long-title Watch Club card, a brick fixture exercising every v1 element, a minimal schema) across themes without creating a room or syncing.
  • Three hard rules — no overlapping text blocks (pairwise getBoundingClientRect intersection), computed font size ≥ 12px on schema text, and titles not clipped (scrollHeight vs clientHeight).
  • On failure — JSON issues plus a screenshot at qa-output/schema-preview-fail.png for the fix loop.

The gate must pass before merging SchemaEngine CSS changes; a manual checklist (SCHEMA_BRICKS_CHECKLIST.md) covers what geometry can't, like vote toggling, owner-vs-member status permissions, and a quick repeat on the Glow and Paper themes. Screenshot + LLM aesthetic review may be layered on later, but the hard rules stay programmatic.

Tradeoffs

  • Expressiveness vs safety — v1 bricks are intentionally few and declarative-only; unknown types are inert data, never executed code, so a malicious or buggy schema can't do anything beyond rendering oddly.
  • No schema editing after create — simpler engine and no migration-of-live-data problem, at the cost of "recreate the room" being the answer to schema mistakes.
  • One shared renderer — declarative rooms all look like "collections of cards"; anything richer (ChoreBoard's payday flow, Trip Split's balances) still earns a builtin template.
