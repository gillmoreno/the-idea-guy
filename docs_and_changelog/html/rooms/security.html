Security & invites

Every Rooms security feature is built client-side: secrets live on your devices and in shared links, never on a server you have to trust.

The big idea: no accounts, no server-side trust

Rooms has no sign-up, no passwords stored on a server, and no user database. Access to a room is access to its room code — a random secret that doubles as the seed for the encryption key. If you have the code, you can decrypt the room; if you don't, the data is just ciphertext. This means the entire security model lives in three places: the links you share, the keys derived on your device, and the local vault that remembers them.

Why this way? Accounts would mean a server that can reset passwords, read metadata, and become a target. By making the room code the only credential, the relay server stays "dumb" — it stores and forwards encrypted blobs it cannot read, and there is nothing valuable to steal from it.

Invite links: secrets in the URL hash

Early versions put the room code in the query string (?c=...). That was a real leak: CDNs, server access logs, and link-preview bots all see and record query strings. Today, every invite parameter lives in the URL fragment (the part after #), which browsers never send over the network.

LinkFormat
Open a room/room#c={roomCode}
Member join/join#c={roomCode}
Admin join/join#c={code}&admin={secret}&template={id}

What users see: an invite link or QR code works exactly the same as before. Behind the scenes, the code never appears in any server or CDN log. Legacy query-string links still parse for backward compatibility, and after a link loads, the app scrubs invite parameters out of the address bar so secrets don't linger in browser history.

Room code + admin secret: two tiers of access

Each room has two encrypted channels with separate keys:

  • Public channel — key derived from the room code alone. Anyone with the member invite can read and write the shared data.
  • Admin channel — key derived from the room code plus a separate admin secret. Admin-only data (settings, controls) physically cannot be decrypted on a member-only device, because the key material simply isn't there.

This is permission-by-cryptography rather than permission-by-policy: there is no server check that could be bypassed, just math.

Optional room passphrase (Argon2id)

The room code alone means a leaked link grants full access. For sensitive rooms, the owner can set a passphrase at creation time and share it out-of-band (in person, over a call). With a passphrase set, key derivation gets meaningfully stronger:

No passphrase (legacy)With passphrase
AES keySHA-256 of the room code materialArgon2id over room code + passphrase
Relay room idSHA-256 of app + scope + codeArgon2id — the ciphertext can't even be fetched without the passphrase

Argon2id is a deliberately slow, memory-hard hash (tuned at 64 MiB / 2 iterations for browsers), which makes brute-forcing a passphrase impractical even if an attacker somehow obtains the room code. The passphrase is stored only in the device vault — never in invite URLs, never on the relay.

Tradeoff: no recovery. Because no server ever sees the passphrase, there is no "forgot passphrase" flow. Lose it and the room is unreadable, period. The UI warns loudly about this at creation time. This is the same deliberate tradeoff made everywhere in Rooms: stronger privacy in exchange for personal responsibility for secrets.

End-to-end encryption: AES-256-GCM

All room data that travels through the relay is encrypted on-device with AES-256-GCM (via the browser's Web Crypto API) before it leaves. Each message gets a fresh random 12-byte IV, and GCM's built-in authentication means tampered ciphertext simply fails to decrypt — there's no way to silently corrupt a room's data in transit.

Device app lock (PIN)

Even with everything encrypted in transit, room codes, admin secrets, and passphrases have to live somewhere on the device — the local vault in localStorage. Anyone who opens your browser could read it. The optional app lock closes this gap:

  • You set a PIN (4+ digits) under Profile → App lock.
  • The entire vault is encrypted with a key derived via Argon2id(PIN, random salt) and the plaintext copy is deleted.
  • On open, the app asks for the PIN, verifies it against an encrypted verifier string, and decrypts the vault into memory for the session.
  • Any vault change while locked is re-encrypted before being saved.

Again, no server and no recovery: forget the PIN and the local copy is gone — but room data isn't lost, because you can rejoin rooms with their invite codes and re-sync from the relay or other devices.

What the relay can and cannot see

The relay seesThe relay cannot see
Opaque room ids (hashes — Argon2id-hardened when a passphrase is set)Room codes, admin secrets, or passphrases
Encrypted blobs and their sizesAny room content — names, tasks, messages, members
Connection timing and IP addresses (like any web server)Who a persona is, or which human is behind a device

The operator of the relay is in the same position as any outside attacker: ciphertext only. This "zero-access" property is structural, not a policy promise.

Threat model in plain words

Protected against: a nosy or compromised relay/CDN operator (sees only ciphertext and opaque ids); leaked server or CDN logs (no secrets in query strings); someone casually opening your browser (app lock encrypts the vault at rest); brute-force on passphrases and PINs (Argon2id is slow and memory-hard); tampering with synced data in transit (GCM authentication).
Not protected against: someone you shared the link with re-sharing it — a leaked room code without a passphrase is full member access; a removed member who keeps the old code (no forward secrecy or key rotation yet, so the public channel stays readable to them); malware or full compromise of an unlocked device; losing all your secrets (no recovery service exists, by design). Server-side 2FA was intentionally skipped because it would require accounts.

Invites between contacts

Beyond shareable links, rooms can be offered directly to mutual contacts (see Personas & contacts). The organizer picks contacts during setup (optional — a soft reminder appears if skipped, and admins can invite later from room settings). Each invitee receives a room_invite message over their end-to-end encrypted pairwise inbox, containing the room code, name, template, and a pre-reserved member slot bound to their persona key. Accepting adds the room to their vault and drops them straight in — no profile-picking step, and the room code never travels in the clear.
