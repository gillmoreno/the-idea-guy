AI generation

Type one sentence on the Create screen, tap ✨ Generate room, and a validated RoomSchema lands in the JSON box — ready to create and share — without any room content ever touching an AI.

What it is

The schema engine already lets anyone build a room from JSON, but the original flow asked users to copy a system prompt into their own AI chat and paste the output back. AI generation removes that round trip: Create → Paste JSON → "Describe it" shows a textarea ("my 4 roommates vote on what to cook this week"), a single button does the AI call, and the result drops into the same JSON box the paste flow uses. The existing validator and create flow take it from there — generation adds an on-ramp, it doesn't add a new path.

The hero moment One plain sentence becomes a working, synced, shareable app in seconds. Everything underneath — validation, the encrypted room document, the invite flow — is the machinery that already existed for pasted schemas.

The privacy boundary

This is the load-bearing decision. Rooms has exactly two backends with two very different trust profiles, and generation never blurs the line:

A generation prompt is an app description the user deliberately sends to an AI ("a chore app where roommates vote") — never end-to-end-encrypted room content. The relay is untouched by all of this and stays dumb. Two AI workloads exist in the plan; this is generation only. The ambient helper (digests that would read room content) is a separate, later spec with its own consent model.

The stateless proxy

genproxy/ is a small separate Go service that exists so the platform AI key never reaches a browser. It holds the key and the system prompt server-side, takes POST /generate with a ≤500-character description, calls the upstream model, strips markdown fences, sanity-checks the shape, and returns {"schema": …}. Full validation stays client-side, same as for pasted JSON.

It is deliberately useless to abusers and cheap to run:

  • Not a free LLM. The system prompt is server-controlled, output is JSON-constrained, and a low max_tokens cap (1600) means there is nothing to extract except room schemas.
  • Per-IP token bucket — 20 generations per hour by default (same mechanism as the relay's limiter; trusts CF-Connecting-IP/X-Forwarded-For because production sits behind a proxy).
  • Hard monthly spend cap — default $10. The proxy's only state is a small spend counter (optionally a JSON file so restarts don't reset the meter). The worst-case AI bill is a chosen number, not a surprise.
  • No database, no logs of descriptions, no stored outputs, no accounts.
Graceful degradation, by contract When the budget is gone, /generate returns 429 budget_exhausted and the app shows "Free generation is resting this month" — pointing at the copy-paste prompt flow, which works with any AI chat, free and unlimited. The same fallback message covers rate limits, network failures, and unusable model output. Generation is an enhancement, never a dependency; the paste-JSON path is the universal fallback underneath.

Free "spells"

Each device gets 15 free generations, counted in localStorage. This is openly honest-majority enforcement in v1 — clearing storage resets it — because the proxy's per-IP and monthly caps are the real rails; the device counter just sets expectations ("12 of 15 free generations left on this device") and nudges heavy users toward the unlimited copy-paste flow. Persona-signed budgets with per-key limits and viral refills (invite friends, earn spells) are the planned v2, once personas ship request signing.

Upstream: swappable by design

Three lanes, picked entirely by env — the rails (budget, rate limits, schema checks) are identical in all of them:

  • GENPROXY_PROVIDER=anthropic (default): the official Anthropic Go SDK → Claude Messages API with claude-haiku-4-5 (~$0.0035 per generation; the $10 budget ≈ 2,800 generations). Needs an Anthropic API key.
  • GENPROXY_PROVIDER=openai: any OpenAI-compatible chat-completions endpoint — e.g. OpenCode Zen with deepseek-v4-flash (~$0.0004 per generation, needs a funded Zen balance).
  • Phase 0, no API billing — the claude-sidecar: genproxy/claude-sidecar/ applies the Inkanto pattern (inkanto/ai-sidecar/): a localhost Node adapter wrapping the Claude Agent SDK, authenticated with CLAUDE_CODE_OAUTH_TOKEN from claude setup-token, drawing from the Claude subscription's monthly Agent SDK credit. It exposes the OpenAI shape on 127.0.0.1:4601, so genproxy points the openai provider at it with zeroed $ prices. Personal-use lane only — swap to a real key for public launch.

On the client, NEXT_PUBLIC_GENPROXY_URL points the PWA at the proxy (defaults to http://localhost:4600 in dev). When it's unset in a prod build, the "Describe it" section is hidden entirely and the panel falls back to copy-paste only — the feature flag is the URL itself.

Behavior users see

  • "Describe it" at the top of the Paste JSON panel: a 3-row textarea (capped at 500 chars), the ✨ Generate room button, and the spell counter.
  • "Conjuring…" while it runs; on success the pretty-printed schema appears in the JSON box and one spell is spent.
  • On any failure — rate limit, exhausted budget, network, garbage output — a short friendly message always routes to the copy-paste prompt below as the no-limits alternative.
  • The copy-paste section ("Or use your own AI chat") and one-tap examples remain right underneath, unchanged.

Tradeoffs

  • Honest device counter — trivially resettable, accepted because the server-side caps bound real cost; signing waits for personas v2.
  • Two prompts to keep in sync — the canonical prompt lives in schema/prompt.ts; the proxy embeds an API-adapted copy (genproxy/prompt.txt). When the schema spec evolves, both must move.
  • No streaming, no in-app BYOK — schemas are small enough that a spinner is fine, and the copy-paste flow already serves power users. Both deliberately out of v1.
  • Hard budget means hard stops — late in a busy month, free generation can be off for everyone. That's the point: the fallback is free, and the bill is bounded.
