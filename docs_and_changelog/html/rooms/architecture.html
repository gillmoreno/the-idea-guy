Rooms architecture

Rooms is a meta-app: one installable PWA shell, one dumb relay server, and many "room templates" (ChoreBoard, Trip Split, Book Club, …) that plug in as views and data schemas — not as separate deploys.

The meta-app idea

Every small group app Gil wants to ship — family chores, trip expenses, a book club tracker — has the same skeleton: people join a shared space with an invite code, everyone's devices keep the data locally, and changes sync between devices through a server that can't read anything. Instead of building and deploying that skeleton over and over, Rooms builds it once and turns each product idea into a template.

The result is a "mini-app factory": shipping Trip Split after ChoreBoard meant adding a folder and a registry entry, not a new deploy, subdomain, relay, or PWA. Marketing can still say "ChoreBoard on Rooms" — the template is the brand, the platform is the plumbing.

One-sentence version: one Docker container at rooms.the-idea-guy.com that serves the PWA and runs the Go relay on the same origin (/sync), with N templates living inside it. A "room" is an invite code plus local device storage — never a subdomain, never a server-side account.

The four layers

LayerWhat it isLives in
ShellCross-template chrome: home (my rooms), create, join, room session, profile, themes, contacts, invites, settings. Owns the device vault, room metadata, and member roles.apps/rooms/web/src/shell/ + routes in src/app/
TemplatesThe actual products. Each template is a React UI plus a slice of CRDT data under its own template.* keys. Mounted by the shell once the room's synced metadata says which template it is.apps/rooms/web/src/templates/
Room kitShared TypeScript library with no UI: crypto (key derivation, encrypt/decrypt), the LocalFirstDoc sync engine, relay protocol framing, invite links, the device vault, personas/contacts, and the single shared Yjs instance.packages/room-kit/
RelayA small Go WebSocket server that broadcasts and replays opaque encrypted blobs per room. No accounts, no auth, no app logic, no ability to decrypt. Shared by every template forever, and shipped in the same container as the web app in production.relay/

The contract between layers is deliberately strict. The shell owns the meta and members maps in each room's CRDT document; templates own only their template.* namespace. The relay knows nothing — not even which template a room runs, because templateId lives inside the encrypted document and the relay room id is just a hash. This keeps every layer replaceable and keeps a misbehaving template from corrupting shell data.

The template registry

A template is a plugin, not a deploy. The registry at apps/rooms/web/src/templates/registry.ts lists each built-in template with an id, name, description, dual-emoji icon, and accent color — currently ChoreBoard, Trip Split, Book Club, Backlog, and Fit Crew. The create flow reads this list, the room page mounts the matching UI when the synced document reveals its templateId, and each template seeds its own template.* CRDT branches.

Two kinds of template exist:

  • Built-in — compiled React code under src/templates/{id}/. Some are flagged ownerOnly (e.g. Backlog) and hidden from public create unless allowlisted.
  • Declarative — JSON-schema-defined rooms rendered by the schema engine, so a new room type can ship without writing React at all.

Adding a built-in template takes three steps: add a registry entry, create src/templates/{id}/ with UI and Yjs data under template.{id}.*, and mount it from the room page when the metadata matches.

Room lifecycle: leaving and deleting

Because a room is just an invite code plus local storage — no server-side account, no central database row — "leaving" and "deleting" need their own local-first definitions:

  • Leave (any member). Leaving clears the session state and navigates straight home, so the action is unambiguous — you're out of the room and looking at your room list, not staring at a stale screen.
  • Delete (admin only). Surfaced as a "Danger zone" in the shared invite/settings block, so every template gets it for free. It's a two-step confirm (Slack-style channel delete) and is disabled while disconnected from the relay — other members can only learn the room is gone through sync.

How deletion works without a server that understands rooms: the admin's client writes a deletedAt tombstone into the room's CRDT meta map, then pushes a storage checkpoint to the relay — the relay's stored backlog for the room collapses to the tombstoned document. The relay stays dumb throughout: it never learns what changed, only that a smaller opaque blob replaced the history. The admin's device then purges its local copy and navigates home.

Every other member's client gates on that tombstone: when deletedAt arrives via sync, the template UI is replaced with a "Room deleted by its organizer" screen offering a one-tap "Remove from this device". Honouring local-first, their copy of the data stays on the device until they choose to remove it, and offline members simply learn of the deletion the next time they sync.

Honest limitation: the admin channel's encrypted blob may remain orphaned on the relay after deletion; idle rooms are evicted from RAM and can expire from disk per relay config, so it ages out rather than being actively reaped.

Why one app instead of many

  • Zero marginal infrastructure per product. No new subdomain, container, relay namespace, or deploy script per template. One static export serves them all; the relay isolates tenants by hashing the room id.
  • Shared hard parts. E2E encryption, offline persistence, invite/role security, themes, personas, and background sync are solved once in the shell and room kit. ChoreBoard's two-channel public/admin security model became the precedent every template inherits.
  • One install for users. A family that uses ChoreBoard already has Trip Split installed — every new template is instantly available on the same home screen, with the same vault and contacts.
  • Cheap to run, low liability. One small container serves static files and moves blobs the relay cannot read. There is no per-app server holding user data (see the Sync & local-first page).

The main tradeoff: all templates ship in one bundle (mitigated by lazy loading) and share one version — a breaking shell change touches every product at once. For a solo builder, that coupling is mostly a feature: one codebase to reason about, one thing to deploy.

One container in production

Rooms used to deploy as two independent pieces — a static site on Cloudflare Pages and a separately deployed relay. That split allowed a real outage: an old relay image speaking a stale protocol against new clients silently broke sync. The fix was structural, not procedural: the web app and the relay now ship as a single Docker bundle, always built from the same commit, so protocol version skew between them is impossible by construction.

Inside the container (deploy/rooms/): supervisord runs two processes — nginx and the Go relay. nginx serves the static Next.js export and proxies exactly two paths to the relay on localhost: /sync (the long-lived WebSocket) and /healthz. The web app's relay URL (NEXT_PUBLIC_RELAY_URL) is baked into the image at build time and points at its own origin — clients sync over wss://rooms.the-idea-guy.com/sync, same-origin, no CORS, no second hostname to misconfigure.

The container is exposed through a cloudflared tunnel. relay.the-idea-guy.com also points at the same bundle so the legacy standalone ChoreBoard app keeps working, and the relay reuses the existing persistent room-data volume (relay_choreboard-relay-data) so no rooms were lost in the migration. Deploying is one command: deploy/rooms/redeploy.sh rebuilds, restarts, and health-checks the bundle.

URLs, ports & dev commands

All routes are client-side in a Next.js 15 static export: / (my rooms), /create, /join, /room#c={roomCode}, plus /profile, /contacts, and /schema. Secrets like admin keys ride in the URL hash so they never reach CDN or server logs. Two newer link conventions follow the same rule: contact "add me" links put the whole contact card in the hash of /contacts (so a phone camera scan opens the app and auto-sends the connection request — the QR code and the copy button share the exact same URL), and member/admin invite links can carry a one-bit pp=1 hash flag signalling the room is passphrase-protected, never the passphrase itself. The /join page can also scan an invite QR with the camera and auto-join on a valid scan, using the shared QRScanner component (generic, filter-by-callback; the contact scanner is a thin wrapper around it).

  • apps/rooms/web/src/templates/ — template plugins (choreboard, tripsplit, bookclub, backlog, fitcrew)
