Rooms architecture

Rooms is a meta-app: one installable PWA shell, one dumb relay server, and many "room templates" (ChoreBoard, Trip Split, Book Club, …) that plug in as views and data schemas — not as separate deploys.

The meta-app idea

Every small group app Gil wants to ship — family chores, trip expenses, a book club tracker — has the same skeleton: people join a shared space with an invite code, everyone's devices keep the data locally, and changes sync between devices through a server that can't read anything. Instead of building and deploying that skeleton over and over, Rooms builds it once and turns each product idea into a template.

The result is a "mini-app factory": shipping Trip Split after ChoreBoard meant adding a folder and a registry entry, not a new Cloudflare project, subdomain, relay, or PWA. Marketing can still say "ChoreBoard on Rooms" — the template is the brand, the platform is the plumbing.

One-sentence version: one PWA at rooms.the-idea-guy.com, one Go relay at relay.the-idea-guy.com, and N templates living inside them. A "room" is an invite code plus local device storage — never a subdomain, never a server-side account.

The four layers

LayerWhat it isLives in
ShellCross-template chrome: home (my rooms), create, join, room session, profile, themes, contacts, invites, settings. Owns the device vault, room metadata, and member roles.apps/rooms/web/src/shell/ + routes in src/app/
TemplatesThe actual products. Each template is a React UI plus a slice of CRDT data under its own template.* keys. Mounted by the shell once the room's synced metadata says which template it is.apps/rooms/web/src/templates/
Room kitShared TypeScript library with no UI: crypto (key derivation, encrypt/decrypt), the LocalFirstDoc sync engine, relay protocol framing, invite links, the device vault, personas/contacts, and the single shared Yjs instance.packages/room-kit/
RelayA small Go WebSocket server that broadcasts and replays opaque encrypted blobs per room. No accounts, no auth, no app logic, no ability to decrypt. Shared by every template forever.relay/

The contract between layers is deliberately strict. The shell owns the meta and members maps in each room's CRDT document; templates own only their template.* namespace. The relay knows nothing — not even which template a room runs, because templateId lives inside the encrypted document and the relay room id is just a hash. This keeps every layer replaceable and keeps a misbehaving template from corrupting shell data.

The template registry

A template is a plugin, not a deploy. The registry at apps/rooms/web/src/templates/registry.ts lists each built-in template with an id, name, description, dual-emoji icon, and accent color — currently ChoreBoard, Trip Split, Book Club, Backlog, and Fit Crew. The create flow reads this list, the room page mounts the matching UI when the synced document reveals its templateId, and each template seeds its own template.* CRDT branches.

Two kinds of template exist:

  • Built-in — compiled React code under src/templates/{id}/. Some are flagged ownerOnly (e.g. Backlog) and hidden from public create unless allowlisted.
  • Declarative — JSON-schema-defined rooms rendered by the schema engine, so a new room type can ship without writing React at all.

Adding a built-in template takes three steps: add a registry entry, create src/templates/{id}/ with UI and Yjs data under template.{id}.*, and mount it from the room page when the metadata matches.

Why one app instead of many

  • Zero marginal infrastructure per product. No new subdomain, Pages project, relay namespace, or deploy script per template. One static export serves them all; the relay isolates tenants by hashing the room id.
  • Shared hard parts. E2E encryption, offline persistence, invite/role security, themes, personas, and background sync are solved once in the shell and room kit. ChoreBoard's two-channel public/admin security model became the precedent every template inherits.
  • One install for users. A family that uses ChoreBoard already has Trip Split installed — every new template is instantly available on the same home screen, with the same vault and contacts.
  • Cheap to run, low liability. Static files on a CDN cost pennies; the relay only moves blobs it cannot read. There is no per-app server holding user data (see the Sync & local-first page).

The main tradeoff: all templates ship in one bundle (mitigated by lazy loading) and share one version — a breaking shell change touches every product at once. For a solo builder, that coupling is mostly a feature: one codebase to reason about, one thing to deploy.

URLs, ports & dev commands

All routes are client-side in a Next.js 15 static export: / (my rooms), /create, /join?code=…, /room?c={roomCode}, plus /profile, /contacts, and /schema. Secrets like admin keys ride in the URL hash so they never reach CDN logs.

ThingWhere
Web app (dev)http://localhost:3300
Relay (dev)ws://localhost:4500
Web app (prod)https://rooms.the-idea-guy.com (Cloudflare Pages)
Relay (prod)wss://relay.the-idea-guy.com (shared, tunneled)

From apps/rooms: make dev starts relay + Next dev with hot reload (make dev-stop kills ports 4500/3300; make dev-relay / make dev-web for two terminals). make preview-all does a production-like static build served locally, with build-time env pointing at the local relay. Deploys: ./scripts/deploy-rooms.sh for the frontend, ./scripts/deploy-relay.sh for the relay.

Key code paths

  • apps/rooms/web/src/shell/ — shell chrome, vault, room session, sync orchestration
  • apps/rooms/web/src/templates/registry.ts — template registry (the one place new templates register)
  • apps/rooms/web/src/templates/ — template plugins (choreboard, tripsplit, bookclub, backlog, fitcrew)
  • apps/rooms/web/src/app/ — client-side routes (home, create, join, room, profile)
  • packages/room-kit/src/ — shared crypto, sync, vault, invites, relay protocol, types
  • relay/main.go — the dumb Go relay
  • apps/rooms/Makefile — dev/preview commands
