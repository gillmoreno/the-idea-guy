Second Brain

A local-first personal notes vault in the Obsidian spirit — rich-text notes with wiki-style links, full-text search, a link graph, AI chat over your own notes, and end-to-end encrypted sync — built as mini-app #3 on the same local-first kit as the Rooms platform.

What it is

Second Brain is a personal knowledge vault that runs entirely in the browser. Notes live in a Yjs CRDT document persisted to IndexedDB, so everything works offline and the app never depends on a server to read or write. You edit notes in a Tiptap rich-text editor, connect them with [[internal links]] that automatically build a backlink index, search everything instantly with MiniSearch, and see how your ideas connect in a D3 force-directed graph driven by the same link index.

Status: not deployed. Second Brain runs locally only. make dev inside secondbrain/ starts the web app on :3200 and its Go relay on :4501. There is no hosted instance yet.

Why it was built this way

The design goal is the same one that drives the Rooms apps: your data should not live on someone else's server in readable form. Second Brain takes the proven room-kit pattern — a Yjs document, IndexedDB persistence, and an encrypted blob relay — and applies it to a single-person (or small shared) notes vault instead of a multi-member room.

  • Local-first, not cloud-first. The vault is the Yjs doc on your device. Sync is optional and additive.
  • The relay is dumb on purpose. It moves opaque encrypted CRDT blobs between devices; it cannot read note content. Operators run sync only.
  • AI is owned by the user, not the operator. API keys live inside the encrypted vault settings, never in relay environment variables.
  • Shared kit, shared lessons. The kit/ sync layer mirrors the room-kit used by ChoreBoard, Trip Split, and friends, so fixes and patterns flow between apps.

What users see

Notes and editing

A sidebar lists folders and notes (with sizes and relative timestamps); the main pane is a Tiptap editor with a toolbar for headings, bold/italic, lists, code, quotes, and callout blocks (info / tip / warning / success — a custom Tiptap node stored as <div data-callout>). Images can be uploaded (max 4 MB), pasted, dragged in, or linked by URL; they are embedded as base64 in the note HTML so they sync encrypted with the vault. Each note's content is its own Y.XmlFragment, edited through Tiptap's collaboration extension, so concurrent edits from two devices merge cleanly.

Links, search, and graph

  • Internal links — type [[ to link another note. Links are stored as <a data-note-id> anchors and indexed in a linkIndex map (outgoing and incoming per note), which powers the backlinks panel.
  • Full-text search — a MiniSearch index is rebuilt on save and reads live text from the Yjs fragments (so search never lags behind the editor).
  • Graph view — a force-directed D3 graph rendered straight from linkIndex, with theme-aware glow.

HTML blocks and HTML pages

Two experimental features let notes break out of rich text:

ScopeOne block inside a normal rich-text noteThe entire note (contentType: "htmlPage")
EditorTiptap chip with HTML + CSS textareas and live previewFull-page preview plus a code pane

Both render inside a sandboxed iframe (sandbox="" — no scripts, no navigation, no form submission to the parent). This was a deliberate choice over raw HTML in the editor DOM (breaks the Tiptap schema, XSS risk) or dangerouslySetInnerHTML (custom CSS would leak into the app UI). The block's HTML and CSS are node attributes that sync over Yjs like any other content, and search/AI tools index the plain text stripped from them.

AI can author full pages: the sidebar's "HTML page" button plus an "AI generate page" flow gives the assistant suggestion chips, lets it read the vault with its tools, then call set_html_page({ html, css, title? }). The same tool works in normal chat — with a note open you can ask "turn this into a pull-up stats dashboard" and the agent converts it to an HTML page.

AI chat with vault tools

The AI panel is an agentic chat that does not guess from random excerpts. It calls read-only tools that run in the browser against the NoteStore and MiniSearch — search_notes (with date filters), get_note, count_notes, list_notes, list_folders, list_tags, get_links, vault_summary — and sends compact JSON results to the model. Answers render as markdown with clickable citation chips that open the cited note while keeping the panel open.

Two providers, both bring-your-own:

  • OpenAI — you paste your own API key in Settings → AI. The key is stored in the vault's encrypted settings map and syncs end-to-end across your devices; the relay never holds it.
  • Ollama (local) — point at a local Ollama server (default http://127.0.0.1:11434); Settings auto-detects installed models via GET /api/tags. No key needed.

When the browser can't reach the provider directly (CORS), the relay's /ai/forward endpoint acts as a dumb pass-through tunnel: it forwards the chat-completion request (including tool_calls) with the user's own Bearer token and holds no server-side API key. Tradeoff acknowledged in the docs: the forward endpoint sees the token in transit over HTTPS, so privacy-maximal users should self-host the relay.

Sync and storage

Sync uses Second Brain's own relay on port 4501 (appId: secondbrain) following the kit pattern: the relay's /sync endpoint shuttles opaque encrypted CRDT blobs, and an invite code lets another device (or person) join the vault. Anyone with the invite code can decrypt the doc — the docs call it "a family secret." Offline state shows a badge; edits queue locally and merge when back online.

Storage is made visible: per-note byte sizes in the sidebar and editor meta row, and a Settings → Storage panel showing vault content total, on-device CRDT size, browser quota, and the largest notes. Because CRDT history can outlive deleted content (especially big inline images), deleting ≥48 KB of image data auto-compacts the doc, and a manual "Compact sync data" button prunes history and pushes a fresh checkpoint to the relay.

Tradeoffs

  • Invite code = full access. One secret both joins and decrypts the vault; there is no per-device revocation.
  • Base64 images bloat the doc. Simple and encrypted-by-default, but storage and sync grow fast — hence the compaction tooling.
  • Sandboxed HTML means no JS widgets. Blocks and pages are static HTML/CSS only; an opt-in allow-scripts mode is a possible future step.
  • BYO AI keys trade convenience for ownership. No server key means zero setup is impossible, but the operator never touches your credentials or note text.
