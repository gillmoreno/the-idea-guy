The Idea Guy website

The personal brand site — a Next.js static front, a small Go content API, and nginx packaged as one Docker container — that turns the repo's experiments into public proof: shipped builds, an idea log, a stealable stack, and snippets.

What it is

"It's time for the idea." The site is the public face of the project: a personal brand and knowledge base exploring what one person can build with AI that's actually useful. It is a single-page Next.js 15 site (static export) whose sections are fed by a tiny Go API reading JSON files — no database, no CMS.

Status: not online yet. The monolith builds and runs locally (docker compose up --build or make dev for Next.js on :3000 + Go on :8081), but it has not been deployed to a public host. The deployment plan is the one-container monolith below.

Why one container

The whole site ships as a single Docker image via Docker Compose. nginx is the only entry point on port 80: / serves the static Next.js export, /api/* reverse-proxies to the Go process on internal :8080, and /health hits the Go health check. One artifact to build, one thing to deploy, no orchestration — the right size for a personal site, at the cost of redeploying the whole image to change content in production.

In local dev the browser talks to Go directly (NEXT_PUBLIC_API_BASE=http://localhost:8081 from frontend/.env.development); in Docker that variable is empty and requests use relative /api/* paths through nginx.

Positioning: the lens

The site's framing is a single repeatable four-step method rather than scattered principles:

  1. Spot the problem — something people actually struggle with today.
  2. Explain how it's solved now — map current solutions with data, charts, and context; show the gaps.
  3. Ship a meaningfully better approach — use modern AI as leverage to build something simpler, more beautiful, more useful, or more powerful.
  4. Teach the process publicly — share the problem, the research, the build, and the lessons.

This lens lives in the hero copy and a #method section on the homepage, with the nav label "The lens." The planned next step is extending each Build entry with problem and betterApproach fields so every shipped item tells the full lens story (problem → how it's solved today → the better approach → lesson).

The content API

Content lives as JSON files in backend/data/ (builds.json, ideas.json, tools.json, snippets.json). The Go server loads them at request time and serves them at /api/builds, /api/ideas, /api/tools, /api/snippets — editing content means editing a JSON file and restarting the server (or rebuilding the image for production). CONTENT_DIR (default data, /app/data in Docker) and API_ADDR (default :8080) are the only knobs.

Tradeoff: dual-maintained content. Each frontend section also carries a hardcoded fallback (e.g. fallbackBuilds in BuildsSection.tsx) used when the API is unreachable, such as during next dev without the Go server. Adding a build means updating both the handler/JSON and the frontend fallback — simple, but two places to keep in sync. Adding a new content type is a five-step recipe: JSON file, loader in store.go, handler in main.go, (covered-by-catch-all) nginx location, and a frontend section.
