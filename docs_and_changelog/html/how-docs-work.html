How these docs work

Reproducible HTML business docs + agent skill + git hooks — same pattern in any repo.

Files

PathPurpose
docs_and_changelog/html/*.htmlHuman-readable business logic
docs_and_changelog/html/doc-manifest.jsonMaps features → source globs → HTML page
.cursor/skills/html-docs/Agent skill: bootstrap + sync
scripts/check-html-docs-sync.shDetects stale docs from git diff
.githooks/post-commitRuns check after each commit

First time (bootstrap)

Ask the agent to run the html-docs skill in bootstrap mode, or say:

“Bootstrap HTML business docs from the manifest.”

It reads the codebase and writes/updates HTML pages to match current behavior.

After code changes (sync)

  1. You commit code as usual.
  2. post-commit hook runs scripts/check-html-docs-sync.sh.
  3. If staged sources changed but HTML did not, it writes .cursor/docs-sync-reminder.json.
  4. Open Cursor and ask: “Sync HTML docs per reminder” (loads html-docs skill, sync mode).

Install git hooks (once per clone)

./scripts/install-git-hooks.sh

Design rules for writers (human or agent)

  • Explain decisions and user-visible behavior, not every function name.
  • One HTML page per feature area; link between pages.
  • When behavior changes, update the matching manifest entry’s HTML in the same PR when possible.
  • Do not duplicate API reference — point to docs_and_changelog/*.md for deep specs if needed.

Optional: run ./scripts/check-html-docs-sync.sh --staged before committing.
