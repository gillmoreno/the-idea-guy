Local-first & relay

Serving the app is cheap; storing family data in the cloud is what we avoid.

Principle

Serve the app centrally, own the data locally. The browser keeps the real copy (IndexedDB via Yjs). The relay only forwards ciphertext.

On each device

  • Yjs — one merged document (CRDT): members, catalog snapshot, completions, permissions, etc.
  • IndexedDB — persists the doc offline.
  • AES-GCM — every update encrypted before it leaves the device.

The relay (Go)

Not a database. A mailbox.

  • WebSocket per room (one room per channel: public vs admin).
  • Append-only log of opaque blobs for catch-up.
  • Optional RELAY_DATA_DIR so restart does not wipe history.
  • Cannot decrypt — never sees chore titles or dollar amounts.

Why not SQL on a server?

  • Provider breach would expose all families’ readable data.
  • Ops/liability for hosting kids’ information.
  • ChoreBoard scale is one family, few devices — CRDT + relay is enough.

Tradeoffs we accept

  • Lost invite code + all devices wiped + empty relay → data gone (mitigate: save codes, 2+ devices, persist relay).
  • Metadata on relay (IPs, timing) still exists.
  • UI permissions are policy, not crypto (see Kid permissions).
