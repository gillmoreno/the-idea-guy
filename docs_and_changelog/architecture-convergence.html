Architecture & Change Safety

One Engine

Rooms quietly grew two architectures. To ship a big bet without fearing every change, they have to become one — every room a template over a single, tested engine.

01The realization: there are two architectures, not one

The mental model — "rooms are like JSONs, plugins made of bricks" — describes an aspiration the codebase only half-delivers. Today Rooms runs two different architectures side by side, and the seam between them is where the confusion (and the bugs) live.

World A — 21 bespoke templates

ChoreBoard, TripSplit, RoomLedger, GameNight… each is hand-written React + its own XyzStore class + its own Yjs shape, wired in by a hardcoded switch in TemplateApp.tsx.

They are isolated from each other (no template imports another) but they are not plugins — they are 21 custom apps sharing a shell.

World B — the declarative engine

A RoomSchema JSON describes collections / fields / features, and one engine (RecordCard, FieldInput, CollectionView) renders it. New room, no new code, no new deploy.

It already has the part that matters: versioning (schemaVersion/engineVersion) and forward-compat — unknown brick types are skipped, never executed.

When you say "plugins made of bricks," you are describing World B. World B is the real bet — the thing that makes "conjure an app in a sentence" possible. World A is the legacy reality you are already, quietly, migrating away from: that is exactly what the /brick-polish loop does — extract a shared brick, delete the per-template copies.

The regression you fear is not hypothetical — it already happened. The "expense card buried the amount mid-sentence across five templates" bug was World A biting you: five hand-rolled money renderers that drifted apart. There was no one place to fix it, so it broke in five.

02The decision: collapse to one engine

Keeping both worlds is the source of the confusion — even for the person who built it. The destination is singular: every room is a template (a schema) rendered by one engine. "Built-in app" stops being an architectural category and becomes, at most, a curated schema we ship in the catalog.

This is not a new initiative bolted onto the roadmap. It is the same line the business plan already draws as its #1 risk — "thin building blocks… the promise breaks the moment someone needs dates, money, RSVPs." Collapsing the two worlds is the richer-bricks roadmap. Each bespoke app is the spec for a brick the engine doesn't have yet.

03The trap to avoid: absorb capability, then delete code

"Make them all templates" is the destination. The fatal way to get there is delete-first-discover-later — ripping out a bespoke app and only then noticing the engine can't express what it did. The order is the opposite: grow the engine until a bespoke app is expressible as pure data, prove it, then delete its code.

Some bespoke apps do things the declarative engine simply cannot say today:

  • ChoreBoard — parent PIN gate, HMAC-signed member completions, dual public/admin Yjs docs, approval workflow. This is agency, not records — the hardest to express declaratively.
  • TripSplit / RoomLedger / CarLog — settlement math (splitMath, balances): computed who-owes-whom, not just stored fields.
  • CarPool / SitCoop — fairness counters (fairness): derived "whose turn" state.
  • GameNight / ScorePad / Bracket — running totals, leaderboards, knockout progression: computed aggregates.

So the real work is a capability gap analysis: classify all 21, and for each, name the engine feature that must exist before it can become data.

TierTemplatesEngine feature the migration requires
EASY bookclub, backlog, whosin, doselog, symptomdiary, cabincal, supperclub, carecircle, carlog (log view) Mostly CRUD lists + votes/status — already expressible, plus a date field and maybe an RSVP feature.
MEDIUM tripsplit, roomledger, coparent, groupfund, sitcoop, carpool A money field + a split/balance feature (computed from records); a fairness/counter feature. These are derived views over records — the engine needs computed fields.
MEDIUM gamenight, scorepad, bracket, fitcrew A number field + a leaderboard/aggregate feature; bracket needs a small progression feature.
HARD choreboard, parts of coparent A roles + approval + gated-agency feature (PIN/secret-bound actions). This is the real frontier — and the proof the engine can hold serious apps. Migrate it last, on purpose.
Every bespoke app you retire is a class of regression you can never have again. A template expressed as a schema has no bespoke code, so it has no bespoke bug surface and nothing to drift. The engine gets tested once; every room built on it inherits that correctness for free.
A few may stay bespoke — and that's a decision, not a default. If ChoreBoard's agency model would distort the engine into something baroque, it is legitimate to keep it as hand-written code. But then it must declare itself: a small, named set of "engine-exempt" templates, each justified, each with its own test coverage — not an open-ended pile of custom apps. The goal is one engine plus a counted, shrinking list of exceptions, not two equal worlds.

04Why one world actually kills the regression fear

The fear is "if I change something shared, I break existing rooms." There are only two ways out, and they are opposites. One is a tax disguised as safety; the other is the grown-up answer.

Isolation (duplication)

A change to TripSplit can't break ChoreBoard. But every bug is fixed N×, quality can't improve globally, and copies drift. This doesn't prevent regressions — it prevents reuse. It's a tax.

Sharing (kit + engine)

Fix once, consistent everywhere. But a change to RecordCard can touch every room at once. Huge blast radius — and that is precisely what makes a change scary.

/brick-polish is deliberately moving you from the first toward the second. That is correct — and it is also why the fear is spiking: you are increasing the blast radius. The answer is not to retreat into isolation. It is to add the thing that makes a big blast radius safe, which today is almost entirely missing.

Isolation doesn't prevent regressions — a contract plus a test net sized to the blast radius does. You already have the contract forming (the kit). What you don't have is the net: zero unit tests, and a single Playwright check on RecordCard CSS. Big blast radius + no net is the genuinely dangerous combination — and the cheap half to fix.

One world is also the honest answer to "manage them independently." Independence is not 21 isolated codebases you can't safely touch — it is N data files over one tested engine, each editable, versionable, and shippable without recompiling anything. The engine is the stable platform; the schemas are the things you manage independently.

05The safety architecture (what makes shared code safe to change)

Five layers, in priority order. The rule running through all of them: the more consumers a piece of code has, the more it must be proven before it ships.

1 · A test net proportional to blast radius

SurfaceWhy it's dangerousTest
Engine core — RecordCard, FieldInput, CollectionView, migrate.tsRuns on user-created schemas in the wild; a bug breaks rooms you've never seenBehavior tests, not just CSS. Render every field/feature type; assert add/edit/vote/status actually work.
Silent-math — splitMath, balances, fairness, permissions, memberSecretsDoesn't look broken — it computes the wrong answerPure-function unit tests. Highest value per test; trivial to write. Do these first.
Kit bricks — Avatar, MoneyAmount, future PersonChip/RecordRow20+ consumers; a styling change ripples everywhereRender/snapshot tests across states × the three themes.

2 · A visual gallery harness (best ROI for a solo dev)

Extend the existing /schema/preview into a storybook-style page that renders every kit brick in every state and every template's main view with seed data, on one scrollable page across all three themes. Change a shared brick → eyeball the entire blast radius in thirty seconds. This is the cheap regression net that a one-person team can actually maintain, and it doubles as the QA surface qa:schema-ui already hooks into.

3 · Treat kit/ + schema/engine/ as a published API

Stable, typed props. Additive by default. A breaking change is a deliberate act: bump a version and migrate every consumer in the same change — which /brick-polish already does ("migrate all consumers, delete the copies"). Formalize that as the rule, not a loop side-effect.

4 · CRDT schema versioning — guard it like crypto

This is the one irreversible surface in the whole product. Encrypted room data lives on users' devices forever, and you can't read it to migrate it server-side. The inviolable rule: never ship a change that reinterprets the meaning of an already-stored field. Only add new optional fields/types; the engine already skips unknowns. Get this wrong after release and you corrupt real data with no rollback. Every other regression in this doc is recoverable; this one is not.

5 · The reuse/duplication line as a written rule

The danger isn't shared code — it's silent sharing without coverage. So the policy: a brick may only enter kit/ once it has render tests; the engine may only gain a field/feature type once it has a behavior test and a gallery fixture. Sharing is earned by coverage, not assumed.

06The one security invariant for a plugin system

The platform boundaries are already right and must stay inviolable: the relay stays dumb, crypto lives in room-kit, invites live in the URL hash. For the plugin dimension specifically, there is exactly one invariant that matters.

A schema is data, never code. The engine interprets schemas; it must never eval them. You already do this — unknown types are skipped, never executed — and it is the entire reason user-made and AI-made rooms can be safe. The moment a "room type" can ship behavior instead of data, you have built a code-injection surface into encrypted rooms that no amount of relay-dumbness can contain. Keep schemas pure data and the security story for plugins — first-party, AI-generated, or third-party — is essentially solved.

This is why convergence and security pull in the same direction. World A's bespoke React is arbitrary code (yours, trusted, but still code that can only be reviewed by hand). World B's schemas are inert data validated against a contract. Every template you move from A to B shrinks the hand-audited surface and grows the mechanically-verifiable one.

07What would be premature (don't build this yet)

A dynamic plugin loader / sandbox

The hardcoded switch + static imports is safer than dynamic loading and completely fine. Don't add a plugin runtime to feel "real." Schemas already give you runtime extensibility without shipping code.

A third-party plugin API / marketplace

Earn first-party stability before third-party extensibility. The engine being schema-driven means you can open it later with almost no new architecture — so there's no cost to waiting.

More isolation

You're already paying the duplication tax. Doubling it to feel safe makes every fix slower and quality worse. The fix is the net, not the walls.

A grand rewrite

Don't stop shipping rooms to "do the architecture." Convergence is incremental: one capability added to the engine, one tier of templates retired, repeat. The app keeps working the whole way.

It is not premature to do the test net and the versioning discipline now. Those pay off immediately and the cost of getting CRDT versioning wrong after release is catastrophic and unrecoverable. Do them before the launch, not after.

08Sequencing

Ordered by leverage. Each step makes the next change less scary; none requires stopping the room factory.

  1. Unit-test the silent-math first — splitMath, balances, fairness, permissions, memberSecrets. Pure functions, days of work, removes the scariest class of invisible regression.
  2. Build the gallery harness — extend /schema/preview to render every kit brick (all states) and every template main view (seed data) across all three themes. Your standing regression net.
  3. Behavior tests for the engine core — RecordCard, FieldInput, CollectionView, migrate.ts. This is the code that runs on rooms you'll never see.
  4. Capability gap analysis — classify all 21 templates (the table in §03), and for each name the engine feature it needs. This produces the brick roadmap directly.
  5. Grow the engine, retire by tier — add date + RSVP, then money + split/balance, then number + leaderboard, then roles/approval. After each, migrate that tier's templates to schemas and delete their bespoke code.
  6. Lock the versioning discipline in writing — the "never reinterpret a stored field" rule, with the additive-only checklist, before launch.
  7. Name the engine-exempt set — whatever stays bespoke (likely just ChoreBoard) becomes a counted, justified, individually-tested list — not an open category.
The one-line bet Don't make plugins safe — make there be only one thing to keep safe. Collapse two architectures into one schema-driven engine, wrap that single engine in a test net sized to its blast radius, and keep schemas as pure data forever. Then "change something without breaking existing rooms" stops being a hope and becomes a property of the system.
