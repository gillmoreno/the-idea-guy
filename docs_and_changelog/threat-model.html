Rooms — Threat Model

An honest account of what Rooms protects, who the adversaries are, what each one can see, and where the real limits are. No marketing, no hand-waving.

01What Rooms protects

The data model is simple: rooms contain CRDT documents. Everything else follows from keeping those documents private to their members.

Room content

Chores, expenses, book notes, workouts — everything inside a room. Encrypted with AES-256-GCM before it ever leaves your device. The relay sees only ciphertext.

Room membership

The relay room ID is a cryptographic hash of your room code. There are no accounts — the server has no membership roster and cannot map IDs back to people.

Invite credentials

Room codes, admin secrets, and passphrases travel in the URL hash fragment — the part after #. Browsers never send the fragment to servers, CDNs, or proxies.

Device vault

All keys and room bookmarks are stored in localStorage on your device. With a PIN enabled, the vault is AES-256-GCM encrypted at rest and is unreadable without the PIN.

Persona identity

Your X25519 keypair lives only in your device vault. Your public key is the contact code you share; your private key never leaves your device and is never sent to the relay.

Contact relationships

Contact data (who knows whom, who blocked whom) is stored locally in the vault. Pairwise messages use ECDH-derived keys; the relay handles them as opaque blobs.

02Cryptography

All primitives are from the Web Crypto API (browser-native, audited) plus hash-wasm for Argon2id in the browser context.

Room channel keys

Without passphrase (legacy, default):key = SHA-256("lfk-key:" + keyMaterial)
With passphrase:key = Argon2id(roomCode + "\0" + passphrase, salt=SHA-256(label))
Argon2id params:memory=64 MiB · iterations=2 · parallelism=1 · output=256 bits
Encryption:AES-256-GCM · 96-bit random IV per ciphertext · 128-bit auth tag

Public vs admin channels

Every room has two encrypted channels. The public channel key is derived from the room code alone — anyone you share the code with can read and write it. The admin channel key is derived from roomCode + adminSecret, a separate 128-bit random secret that only the room creator holds. Admin-only data (settings, moderator controls) is physically unreadable on member-only devices — not just hidden behind a UI check.

Public key material:"lfk-public:{roomCode}"
Admin key material:"lfk-admin:{roomCode}:{adminSecret}"

Relay room IDs

The relay never receives a room code. The ID it stores is a derived value:

Without passphrase:SHA-256("lfk-room:{appId}:{scope}:{roomCode}") → first 16 bytes as hex
With passphrase:Argon2id(roomCode + "\0" + passphrase, ...) → the relay room ID itself requires the passphrase to compute; data cannot be fetched without it

Device PIN (vault lock)

Lock key:Argon2id(PIN, salt=random 128-bit hex) → 256-bit AES-GCM key
Storage:rooms.vault.v1.cipher — AES-256-GCM ciphertext of vault JSON
Verification:Separate AES-GCM ciphertext of a known plaintext — wrong PIN → nil plaintext, no oracle

Biometric unlock (WebAuthn PRF)

Credential:Platform authenticator (Face ID / Touch ID / Windows Hello), user verification required, attestation none
Unlock key:HKDF-SHA256(PRF output, info="lfk-vault-bio:v1") → 256-bit AES-GCM key that wraps the raw Argon2id PIN-key bytes
Guarantee:Biometric and PIN decrypt the same vault ciphertext — no second copy of the vault, nothing sent to any server. PIN always remains as fallback; changing the PIN invalidates the wrap and requires re-enrollment

Persona keypairs

Algorithm:X25519 (ECDH key agreement)
Pairwise inbox key:AES-256-GCM, key = ECDH(myPrivate, theirPublic)
Private key:Stored only in device vault — never transmitted
Implementation source

All crypto logic is in packages/room-kit/src/crypto.ts. It uses crypto.subtle (W3C Web Crypto API) exclusively — no third-party crypto libraries for the symmetric layer. Argon2id via hash-wasm (WASM, audited by Cure53 in 2021).

04Relay operator

The relay is intentionally "dumb." It was designed with the constraint that even a malicious operator should learn as little as possible about your rooms.

What the relay does see

  • IP addresses of connecting clients (unavoidable with TCP). Use a VPN or Tor to hide your IP from the relay.
  • Derived room IDs — hex strings that are SHA-256 or Argon2id hashes of the actual room code. They reveal nothing about room content or purpose.
  • Which IPs share a room ID — so the operator can infer that two IP addresses are in the same room (even without knowing what the room is).
  • Blob sizes and timing — the size of each sync frame and when it was sent. A very active room produces larger, more frequent blobs.
  • Connection counts — how many concurrent connections a room has.

What the relay cannot see

  • Room content. Every frame is AES-256-GCM ciphertext. The relay stores and forwards opaque bytes. Source: relay/main.go — "It can never read the contents."
  • Room codes or passphrases. These never travel to the relay. The relay ID is a one-way hash.
  • Member identities. No accounts, no usernames, no profiles are registered on the server.
  • Which room belongs to which person. Room IDs are hashes; without the pre-image (room code + passphrase) they cannot be reversed.
Social graph leakage

Even though the relay cannot read room data, it can observe that IP address A and IP address B both connect to the same hashed room ID. Over time this reveals a social graph: "these people are in the same room." This is inherent to any server-based sync architecture. Mitigation: use the same network (home Wi-Fi, VPN, or Tor) for all members, or accept this as a known limitation.

Why we can trust the relay's blindness

The relay's inability to read content is not a matter of policy — it is physically enforced by the encryption. Even if the relay operator wanted to read your room, they cannot, because they do not have the AES key. The key is derived from your room code (which lives only on your devices, never on the server).

05Network observer

An attacker who can see your internet traffic (ISP, coffee shop Wi-Fi, corporate proxy) is blocked by two independent encryption layers.

  • TLS 1.3 encrypts the connection to the relay. The observer sees a stream of encrypted bytes going to a server; they do not see room IDs, blob contents, or anything else.
  • AES-256-GCM encrypts the room data itself. Even if TLS were somehow broken, the inner layer is intact.
  • The URL hash (room code, admin secret, passphrase) is never sent to any server. Browsers strip the hash before making requests. A network observer cannot capture your invite credentials from traffic.
Invite link safety on the wire

Sharing a link like rooms.example.com/join#c=abc123 — the #c=abc123 fragment never appears in any HTTP request. CDN logs, server access logs, and link-preview bots see only /join.

06Physical device access

The most dangerous adversary is one who physically has your unlocked device. Here is exactly what that means.

Without PIN lock (default)

Your vault is in plain localStorage

By default, room bookmarks, invite codes, admin secrets, and your persona private key are stored as plaintext JSON in localStorage under the key rooms.vault.v1. Anyone who can open your browser's DevTools or access the browser profile on disk can read everything.

  • IndexedDB stores the CRDT blobs. The blobs are encrypted (AES-256-GCM), but the key is derivable from the room code which is in the plain vault — so a device attacker can decrypt the blobs too.
  • This is the same threat model as any password manager that doesn't require a master password to open.

With PIN lock enabled

Vault becomes ciphertext

When you set a PIN (Profile → App lock), the vault JSON is encrypted with AES-256-GCM using a key derived via Argon2id from your PIN and a random salt. The plaintext vault key (rooms.vault.v1) is deleted. Only the ciphertext (rooms.vault.v1.cipher) and the lock metadata remain.

  • Brute-force cost: Argon2id with 64 MiB memory and 2 iterations takes ~200–800 ms per attempt on a phone-class device. A 6-digit PIN has 1,000,000 combinations → worst-case ~200,000 seconds (~2.3 days) at one attempt per 200 ms. A passphrase is much stronger.
  • No recovery. We do not hold a copy of your PIN or vault. Forget the PIN → rejoin rooms with their invite codes. This is by design: recovery requires a server that stores secrets, which undermines the privacy model.
  • Session memory. While the app is open and unlocked, the decrypted vault lives in JavaScript memory. A browser crash dump or memory forensics tool could potentially extract it. Close the browser when not in use. (The PIN itself is not retained after unlock — only the derived AES key, which is needed to re-encrypt the vault on save.)
  • Biometric unlock is convenience, not a stronger key. Enrolling Face ID / Touch ID lets the platform authenticator release a secret that unwraps the same vault key your PIN derives. It resists shoulder-surfing and is faster, but anyone who knows the PIN can still unlock — and enrollment is per device. The biometric template never leaves the platform's secure hardware; Rooms only ever sees the derived secret.

Remote wipe

There is no remote wipe feature. If your device is stolen, the right action is to rotate room codes on another device (which generates a new room code and effectively locks out the old device — see Honest limits for caveats).

07Room members & ex-members

The security model treats the room code as the credential. Anyone who holds the code can read and write the public channel — including people who were "removed."

Public channel (room code)

  • Any device with the room code can join the public channel. There is no server-side membership check — by design (no server accounts).
  • Removing a member in the UI deletes their CRDT member entry from the document. It does not change the room code. The removed person can still read new data if they kept the code.
  • To truly revoke access: the room admin must generate a new room code. New members get the new code; the old member's code becomes stale. This requires everyone to rejoin. The UI for this is planned but not yet in v1.

Admin channel (admin secret)

  • Admin-only data is encrypted with a separate key that includes the adminSecret. A member-only device physically cannot decrypt admin blobs — the key material is never on their device.
  • Removing someone from admin requires rotating the admin secret (same migration as rotating the room code).

Passphrase rooms

Extra layer against code leaks

With an optional passphrase set, the AES key and even the relay room ID are Argon2id-derived from both the room code and the passphrase. Someone who obtains only the room code cannot decrypt the blobs — they need the passphrase too, which travels out-of-band (never in the invite URL).

08AI generation proxy (genproxy)

The "speak it into existence" feature lets you describe a room in plain English. Here is what the proxy sees and what it does not.

Stateless — room data never touches it

The genproxy receives only the text you type in the "Describe it" box (e.g. "a weekly chore chart for 3 kids and a dog"). It returns a JSON schema. It never sees your room contents, member list, vault, or any existing data.

  • What it receives: Your description prompt, device spell counter, and app version. Nothing else.
  • What it sends to the AI model (Anthropic): The prompt text and a system prompt. No room data, no keys, no personal information beyond the text you wrote.
  • No storage: The proxy is stateless and keeps no logs of prompts. Each request is independent.
  • Anthropic's data practices apply to the prompt text itself. If your description contains sensitive names or details, consider using a generic description instead.
  • Copy-paste fallback: If you prefer zero-server contact, the UI provides a copy-paste path — copy the system prompt, paste into any LLM you trust, paste the result back. No network call to the proxy.
The proxy never touches your rooms

The genproxy only generates the initial schema. It has no access to the relay, no room codes, and no ability to read or write room data. Once you create the room, all sync goes directly device ↔ relay (encrypted).

09Honest limits

Every security design has limits. Here are ours, stated plainly. We think trust is built by naming what we don't do, not by overselling what we do.

No forward secrecy

The room code is a static long-term secret. If it leaks in the future, historical ciphertext stored on the relay (up to the log compaction point) can potentially be decrypted. Passphrase rooms mitigate this significantly (Argon2id derivation means brute-force on the passphrase is expensive). A future version will add ephemeral per-session keys (Double Ratchet or similar).

No member revocation (v1)

Removing a member from the room UI does not change the encryption key. The removed member can still connect to the relay and read new public-channel data using the same code. True revocation requires rotating the room code — everyone rejoins, the old code stops working. v1 does not have a UI for this yet. It's the #1 planned security enhancement.

No CRDT content signatures (v1)

Any device with the room code can write CRDT entries with any member ID. There is no per-member signing key in v1 to prove authorship. A malicious member could write data attributed to another member. Planned: each member slot will carry a signing keypair; writes will be signed and verifiable.

SHA-256 key derivation without passphrase

Rooms without a passphrase derive the AES key via SHA-256 — fast and non-iterated. This is secure against the relay (which has only the ciphertext) but provides no brute-force resistance if someone obtains the room code and is trying to guess it via dictionary attack. Room codes are 128-bit random — not guessable in practice. For sensitive rooms, add a passphrase.

Browser storage model

localStorage is origin-scoped but not process-isolated. Browser extensions with broad permissions, XSS vulnerabilities in the app, or shared browser profiles can access vault data. PIN lock encrypts the vault but JavaScript executing on the same origin can still read it from memory while the session is active. This is a fundamental web-platform limitation — a native app would have stronger isolation.

Relay IP metadata

The relay logs connection events including IP addresses. Even with E2E encryption, the relay operator can build a social graph from IP co-location (who connects to the same hashed room). This metadata can be as sensitive as content. Use a VPN or Tor to prevent IP disclosure if this matters to you.

No independent audit (yet)

The cryptographic implementation has not been reviewed by an independent security firm. The primitives are standard (AES-256-GCM, Argon2id, X25519), but the key derivation scheme, vault format, and relay protocol have only been reviewed internally. An external audit is planned before any use case with high-stakes data.

10Summary: the one-page version

What is true today

  • Room data is AES-256-GCM encrypted on your device before sync
  • The relay stores and moves only opaque ciphertext — it physically cannot read your data
  • Invite codes live only in the URL hash — servers, CDNs, and proxies never see them
  • Admin data uses a separate key that never touches member-only devices
  • Passphrase rooms use Argon2id — expensive to brute-force even with the room code
  • PIN lock (opt-in) protects the vault at rest with Argon2id key derivation
  • Persona private keys never leave your device
  • No accounts, no passwords, no password-reset emails, no central identity store

What is not true (yet)

  • No forward secrecy — room code is a static long-term credential
  • No member revocation without rotating the room code manually
  • No per-member content signatures — authorship not cryptographically provable in v1
  • No remote wipe
  • No independent security audit yet
  • IP addresses visible to relay operator
  • Without PIN, vault is plain localStorage — open to anyone with device access
The honest bottom line

Rooms is meaningfully more private than cloud alternatives for the threat models that matter most to most people: a curious platform, a nosy server admin, a link leak to the wrong person, or someone grabbing the invite URL from a CDN log. It is not a tool for journalists in adversarial environments or anyone whose threat model includes a device seizure or a subpoena to the relay operator. For those cases: no audit yet, no forward secrecy, no revocation — wait for v2 of the security layer.
